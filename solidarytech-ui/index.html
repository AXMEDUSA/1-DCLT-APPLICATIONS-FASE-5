Gerencia o cadastro e consulta de ONGs parceiras da plataforma. Cada ONG possui nome, e-mail único, causa e cidade.

GET
/health
Verifica se o serviço está ativo

/ngos
Cadastra nova ONG

/ngos

Go 1.24 · PostgreSQL + Queue · Porta 8082

Caminho crítico da plataforma. Processa doações em tempo real, persiste no PostgreSQL e emite eventos na Azure Storage Queue. SLO: 99,9% sucesso · P99 ≤ 500ms

/donations

volunteer-service
Python / Flask · CosmosDB Table API · Porta 8083

Match entre voluntários e ONGs. Cada voluntário recebe um UUID único e é associado a uma ONG pelo ID. Armazenamento no Azure CosmosDB (Table API, serverless).

POST
/volunteers
Registra novo voluntário
GET
/volunteers/<ngo_id>
Lista voluntários de uma ONG

Python 3 / Flask · CosmosDB Table API (Serverless)
Endpoints
GET
/health
Health check

/volunteers
Registra voluntário (UUID gerado automaticamente)
Body: {"name":"...","email":"...","ngo_id":1}

Recursos no cluster
🔹 Banco: Azure CosmosDB Table API (Serverless)
🔹 SDK: azure-data-tables (migrado de boto3/DynamoDB)
🔹 CPU: request 100m / limit 250m
🔹 Memória: request 128Mi / limit 256Mi

Como rodar localmente
kubectl port-forward
Comandos port-forward
# Terminal 1 — ngo-service
kubectl port-forward svc/ngo-service -n solidarytech 8081:80
# Terminal 2 — donation-service
kubectl port-forward svc/donation-service -n solidarytech 8082:80
# Terminal 3 — volunteer-service
kubectl port-forward svc/volunteer-service -n solidarytech 18083:80
Arquitetura de dados
🗄️ ngo-service → PostgreSQL pg-ngo-solidarytech-v2
🗄️ donation-service → PostgreSQL pg-donation-solidarytech-v2
📨 donation-service → Azure Storage Queue fila-solidarytech
🌐 volunteer-service → CosmosDB cosmos-solidarytech-f5
